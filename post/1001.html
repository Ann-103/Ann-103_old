Chapter 1 概述

笔记参考中科大李卫海教授的ppt，参考书为《编译原理》陈意云。实验部分是Tinyc的学习和修改，参考实验文档链接。初次学习，如有错误，还望指正，谢谢。

概述：编译器的目标是将源程序翻译成可执行的机器代码，左侧为前端部分，右侧为后端。

遍：将源程序输入对应模块得到输出。通常前端会做“一”遍，在词法分析的同时做语法分析等（不会完全串行），后端可能会多种优化，做多遍。

解释器和编译器的补充
类别描述对象运行时机应用语言中间代码速度错误
解释器（火锅）高级到低级语言，只有前端四个过程程序整体编译程序运行前编程语言，如C、c++、c#、Scala有快同时显示所有错误，难，出现错误后停止，后从头编译
编译器（家常菜）高级程序语句转换成机器代码，边解释边执行逐行编译程序运行时PHP、Ruby、Python无慢出现错误则阻止翻译，删除后继续

一般我们认为的低级语言有汇编（助记符）和机器语言等。

解释器的优化过程困难，执行效率低，前端部分需反复分析

实验的部分说明

tinyc是一个简单的c语言编译器

  • 所有 C 语法的规则均适用于 TinyC 语法，
    • 语句有赋值，函数调用，if和while。
    • 支持算符：+, -, *, /, %, ==, !=, >, <, >=, <=, &&, ||, !, -
    • 不支持 ++ 和 - -
    • 赋值语句只能单独使用，不能放在表达式内部
  • 只有 int 一种数据类型（32位）
  • 函数的返回值可以声明为 int 和 void 两种类型，但编译器会自动为 void 函数返回一个 int 值
  • 不支持全局变量，只有局部变量，变量须先声明再使用，且变量声明必须放在函数体的最前面，不支持声明变量的时候赋初值。
  • 不支持函数原型声明，函数声明必须和定义在一起，函数无需先定义再使用
  • 整个程序必须有一个不带参数的 main 函数，此为程序的入口。
  • 不支持 #include 等预处理命令，不支持多行注释。

步骤简介

  • 前端

    • 词法分析（线性分析/扫描）：源文件的字符流拆分分一个个的词（token）

      • 字符流中声明过的变量等存储在符号表中，识别变量、符号等并进行划分，输出记号流。

    • 语法分析 （分析）：构造语法树

      • 检查记号流是否符合编程的语法规定，并用表示为树形的中间表，如语法树。

    • 语义分析

      • 简单来说除了纠错外会对语法树升级添加部分需要做的操作。

    • 中间代码生成器

      • 语法树 -> 三地址中间代码

      产生更低级的显式中间表示，这种中间表示想象成一种抽象机的程序，三地址中间代码是其中一种表示，每个指令最多有三个操作数，除了赋值算符以外最多有一个算符，在生成时编译器需要产生临时变量名，决定运算优先顺序等。

  • 后端

    • 代码优化器：独立于机器的代码优化阶段改进中间代码产生较好的目标代码（仍然是中间代码），例子中对于变量数量和类型转化也做了优化。
    • 代码生成器：将三地址中间代码转化为汇编代码。
编译技术的应用

高级语言的实现

  • 高级语言易于编程但是运行速度较慢，和低级语言相反，低级语言难以编写、易出错、难维护。
  • 流行语言的演变向抽象的方向进行，同时刺激着编译器的优化。

计算机结构体系的优化

  • 计算机体系结构迅速演化：需要并行接纳多处理器、使内存分层更有效而非集中在优化处理器执行
  • 计算机性能提升需要编译器充分利用其特征代码。比如RISC是编译器计数影响计算机体系结构
  • 二进制翻译：翻译不同机器码执行。
  • 数据库查询解释器。

改进软件开发效率

  • 类型检查：是一种捕捉程序中前后不一致的成熟而有效的技术
  • 边界检查：数据流分析技术可用来定位缓冲区溢出
  • 内存管理：自动的内存管理删除内存泄漏等内存管理错误

语言和文法

文法的优点

  • 精确，易理解的语法说明

  • 自动产生高效的分析器

  • 语言定义出层次结构

  • 便于修改

文法的问题

  • 文法只能描述编程语言的大部分语法，不能描述语言中上下文有关的语法特征

文法比正规式更强，但是在词法分析阶段还是使用正规式，词法规则非常简单，不必用文法，对于词法记号，正规式描述简洁且易于理解。从正规式构造出的词法分析器效率高

正规式和上下文无关文法的比较

  • 正规式描述的都可以用用上下文无关文法**描述

比如：正规式(a|b)*ab 构造的文法：

A0 → aA0 | bA0 | aA1
A1 → bA2
A2 → ε

  • 将有限自动机变换为上下文无关语法，对每个状态引入非终结符Ai，然后如果有一个状态转化，引入Ai → aAj，如果是空转换，引入Ai → Aj，如果i是接受状态（终态），引入 Ai → ε 。

也可以直接 A0 → aA0 | bA0 | ab ,

  • 合在一起需要反省语言的注解和空白等，复杂。

预测分析的错误恢复

(1) 先对编译器的错误处理做一个概述

  • 词法错误，如标识符、关键字或算符的拼写错
  • 语法错误，如算术表达式的括号不配对
  • 语义错误，如算符作用于不相容的运算对象
  • 逻辑错误，如无穷的递归调用

(2) 分析器对错误处理的基本目标

  • 清楚而准确地报告错误的出现，并尽量少出现伪错误
  • 迅速地从每个错误中恢复过来，以便诊断后面的错误
  • 它不应该使正确程序的处理速度降低太多

(3) 非递归预测分析在什么场合下发现错误

  • 栈顶是终结符，但和输入符号不匹配
  • 栈顶是非终结符A，输入符号是a，但M[A , a]是空白

(4) 非递归预测分析：采用紧急方式的错误恢复

  • 发现错误时，分析器每次抛弃一个输入记号，直到输入记号
    属于某个指定的同步记号集合为止

(5) 同步

  • 把**FOLLOW(A)**的所有终结符放入非终结符A的同步记号集合

概念 解释 补充

活动表的原则：之前介绍的顺序就是满足这些原则的一个结果，但是并不完全一样

  • 中间某个位置是基地址，向上增加元素，但是由于地址从高往低，即减去相应的偏移
  • 长度能较早确定的域放在活动记录的中间
  • 一般把临时数据域放在局部数据域的后面。但是现在局部变量的声明可以在中间，故可能这个原则不会完全成立。
  • 把参数域和可能有的返回值域放在紧靠调用者活动记录的地方（头头），并没有特别的顺序
  • 用同样的代码来执行各个活动的保存和恢复 ，即生成这个活动表的代码，最好都一样

    • 不会试图得到每个点的完整状态或到达路径，而是抽取出特定数据流需要的信息（目的），总结出有限的事实
    • 事实和路径无关，仅与程序点有关
