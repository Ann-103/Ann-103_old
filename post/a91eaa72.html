本文是针对git的工作原理和操作为主的学习记录,参考资料：《github入门与实践》，参考网站：https://www.liaoxuefeng.com

GIT概述

Git是一个开源的分布式版本控制系统。可以在本地建立库，控制项目的修改版本，也可以拷贝给别人。对多人合作项目，项目备份和版本回溯有要求的，可以使用git 将源码托管到代码托管平台上（Github，Gitee，Gitlab）。个人项目（本地）可以和github上的仓库建立连接，他人可以使用git将源码从平台克隆到本地，本地开发者对源代码修改后，通过pull request功能申请合并到托管的仓库，并记录更改前后差别。

Git的版本管理功能类似于撤销回滚，

分布式控制系统是相对于集中式而言的，集中式将所有的数据都放在服务器里，开发者需要联网连接到服务器才能修改数据，而git中，每个人电脑上都是完整的版本库。

Git安装与配置

安装

在 Windows 环境中，最简单快捷的方法是使用 msysGit ，登录官网https://git-for-windows.github.io/下载，按照 Downloads 的向导下载安装包，然后选择组件、设置环境变量等。Git Bash 会作为应用程序添加进系统，有关git的命令我们在这个窗口进行，安装过程中会有一个默认”Use Git from the Windows Command Prompt“的选项，表示命令在window的cmd命令行中也可以运行。

建议搜索将gitbash集成在windows终端的方法，美观且方便复制粘贴

配置

  • 在命令行设置姓名和邮箱地址，这里的信息对应了电脑提交的署名

  • 命令生成~/.gitconfig文件，可以设置

    从而让命令的输出有更高可读性

电脑连接

建立电脑与你的github账户的连接，这样电脑就有权限向你的github账户推送

  • 生成ssh key，解决本地和服务器的链接问题，首先打开电脑文件夹，找到C:\Users\名字.ssh文件夹并删除，git bash:

    输入三次回车

    打开.ssh文件夹，找到.ssh\id_rsa.pub文件，记事本打开并复制里面的内容

  • 打开github主页，进入setting-> access->ssh and gpg keys，把复制的内容粘贴进去，自己命名保存

    输入ssh -T git@github.com验证，如果崩出来个选项输入yes回车，如果返回Hi！ ，You’ve…即成功。

Git仓库的初始化

本地仓库的使用

  • 本地库初始化 git init，新建或选择一个文件夹作为工作区

    如果在windows中使用wsl的话也是可以直接进行git的操作的

  • 初始化完成后，根目录会多出来一个.git的文件夹，称之为版本库，包含了暂存区，以及git自动创建的master分支，这个文件夹不需要人为修改

    • git支持纯文本文件的改动追踪，micro的word文件使用二进制格式，故只能进行回滚撤销，不知道具体增加减少的文本，建议使用vs或sublime等编辑txt文件

  • 在文件夹中右键gitbash黑窗，分别使用add，commit命令可以将文件添加到本地仓库中

    • $ git add readme.txt //表示将文件加入暂存区
$ git commit -m "wrote a readme file"//表示将暂存区的文件推送到本地仓库

  • 取消回滚，即不小心滚到错误版本

    • $ git log//查询提交历史，自己看需要回到那个版本，commit后会有很长的一个标识串
$ git reset --hard 1094a//取标识串前几位自动匹配

命令操作

  1. 提交

    • 对本地文件进行修改，工作区文件添加到暂存区，再提交

    • 提交到本地仓库

  2. 回退

    • 查询log，然后看需要回退多少步

      因为Git在内部有个指向当前版本的HEAD指针，相当于指针的更改

  3. 编辑

    • 暂存区文件删除

      暂存区文件删除，同时删除工作区文件

      取消暂存

    • 未上传时撤销对文件的更改，回到最近一次git commit或git add时的状态

      同时，如果误删了工作区的文件，使用这个操作可以直接把某个文件恢复，前提是版本库中有这个文件的其他版本，如果没有-- 代表进入某个分支、

    • 隐藏当前变更，以便能够切换分支

    • 重命名文件，并将已改名文件提交到暂存区

  4. 查询

    • git status 查看当前分支状态，文件是否被修改，甚至可以提示新增的文件如何删去等操作

    • 工作区中，已修改未提交的文件对比

    • 比较仓库和暂存区的不同

    • 比较暂存区与上一版本的差异

    • 查看历史记录：从近到远

      commit id每个人都不一样，加密后是sha数字

    • 记录每一次命令，用来查找被撤销的新操作id等

远程仓库

本地推送到远程

  • 申请github账号并创建repo仓库，将自己本地的仓库与之关联，本地运行

    只有自己的库才有推送的权限，即电脑的ssh key被github账号认证

  • 把本地库的所有内容推送到远程库上

    由于远程库是空的，我们第一次推送master分支时，加上了-u参数，Git还会把本地的master分支和远程的master分支关联起来，以后不用

  • 解绑

远程克隆到本地

从github克隆一个本地库，多人协作开发则人均克隆一份即可。在github上也提供了诸如https的地址，但是每次推送都需要输入口令，比较麻烦，具体使用需要看情况

  • 分支：不是开叉哦，是时间上的并行，对其他分支透明的进度更改，可以将两个分支合并

  • 主分支即master分支，HEAD指向当前分支，比如我们切换到了dev分支并进行推送，则HEAD会跟着dev移动

    • 首先，我们创建dev分支，然后切换到dev分支

    • 查看当前分支

    • 在分支上提交

    • 合并指定分支到当前分支，直接把master指针指向dev的当前提交节点，换句话说，更新master主分支的状态更新为dev的当前状态。

  • 如果不同分支不在同一时间轴上，即并没有在分支修改完后直接合并，而是修改了另外一个分支的内容导致分叉

$ git switch master

$ git switch master
$ git merge --no-ff -m "merged bug fix 101" issue-101

$ git switch dev
$ git stash pop//恢复存储，会把stash内容删除，或者使用git stash apply和git stash drop分别恢复和删除

$ git stash list
$ git stash apply stash@{0}//先查看后恢复指定stash

$ git cherry-pick 4c805e2//这里调用日志查询fix101的状态对应值是4c805e2,cherry命令重放了bug修复的提交，同理可以直接在dev上修改bug后重放到master上，不过如果依旧无法提交dev工作区部分内容，还是需要借助stash

  • 多人协作

    • 要查看远程库的信息，如果没有推送权限，就看不到push的地址

    • 推送分支，就是把该分支上的所有本地提交推送到远程库

    • 抓取分支

    • 冲突，比如同时抓取了dev并修改，但是无法两个人都push因为状态改变，可以再次pull下来后解决冲突再发送。

      git pull也失败了，原因是没有指定本地dev分支与远程origin/dev分支的链接，根据提示，设置dev和origin/dev的链接：
