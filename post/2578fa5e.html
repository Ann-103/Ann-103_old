Compilation, Linking and Loading of Programs 程序的编译、链接和装入

地址空间

  • 程序装载到内存才可以运行，一般保存在磁盘（.exe)
  • 多道程序设计模型，运行程序需建立进程，首先分配内存 ，允许多个程序同时进入内存
  • 每个进程有自己的地址空间

内存地址

  • 逻辑地址：相对，虚拟地址。编译、汇编后形成目标代码 .o为相对首地址编址

  • 物理地址：绝对，实地址。内存中存储单元的地址 可直接寻址

程序装入内存

装入方式

静态重定位：进程装入内存时，一次性完成到物理地址PA的转换，运行时不再进行，可以由软件完成

动态重定位：执行到对应语句时才完成到 PA 的转换，需要硬件支持（修改基址寄存器）

重定位：在装入时对目标程序中指令和数据的修改过程，

  • 编译程序产生绝对地址的目标代码，也可由程序员直接赋予。通常程序员宁可在程序中采用符号地址，然后在
    编译或汇编时，再将这些符号地址转换为绝对地址。 （变量）

  • 装入内存后不需对程序和数据的地址进行修改

多道程序环境下，不可能预知目标模块放在内存中的地址，因此绝对装入方式不适合在多道环境下使用

Memory Partitioning 内存分配

分区是指内存中的一个连续区域。

概要

  • 连续（分区管理方式）：连续的内存空间
    • 固定（定长或静态分区分配）：Fixed partition: shelving the boxes 最简单的可运行多道程序的内存管理方式
      • Equal-size partitions：过大浪费，过小不够
      • Unequal-size partitions ：根据程序大小决定所使用的分区
    • 动态: Dynamic Partition: stacking the boxes
    • 伙伴：Buddy System: splitting & merging the shelves
    • 单一连续分配：单用户、单任务 OS，一个程序一个连续内存空间
  • 非连续：一个进程分散地装入到不相邻接的分区
    • 页式管理
    • 段式管理

分区管理方式

内存分为系统区和用户区，一般情况下无存储器保护机构

  • 系统区：仅提供给OS使用，通常是放在内存的低址部分；
  • 用户区：除系统区以外的全部内存空间，提供给用户使用

特点：简单易行，系统开销小，资源利用率低(一次只能装入一个作业)

描述：静态地把内存用户空间划分为若干个固定大小的连续区域，每个分区中只装入一道进程

内存分配表(分区说明表)：分区号、分区大小、起始地址和分区的使用状态。内存的分配释放、存储保护以及地址变换都通过分区说明表进行

进程表：进程名、进程大小、进程占用的分区号

过程： 内存分配程序检索表分配或拒绝

  • Equal-size partitions ：无特殊，每个分区都需要维护一个调度队列
  • Unequal-size partitions ：
    • per-partition queue ：减少内部碎片，单个分区最优，等待，小内存的进程可能被阻塞即使有大的空闲分区
    • global queue: 所有的进程只提供一个队列，有都被占据，则必须进行交换，换出阻塞进程等

优缺点：

  • 优：进程在执行时，不会改变存储区域，所以采用静态地址重定位方式。此方法易于实现，系统开销小
  • 缺：分区的数量在系统生成阶段已经确定，因而 限制了系统中并发进程的数量
  • 缺：分区的大小是在系统生成阶段实现设置的，因而 小作业不能有效地利用分区空间
  • 分区长度和数量是可变的，动态分区方法会在内存中形成许多小空洞（外部碎片），内存利用率随之下降。
  • 克服外部碎片的一种方法是压缩，非常耗时且需要动态重定位的能力
概念 解释 补充
内部碎片 一个存储分区内
外部碎片 通常指一个暂时不能使用的存储分区
分配分区表 描述已被分配的区域
空闲分区表 描述内存中的自由区域 一般的实现是一条空闲分区链
空闲分区链 为每一个自由分区设置一个链接指针来指向下一个自由分区，使所有的自由分区形成一个链表。 含头指针

First-fit > Next-fit > Best-fit

放置算法 描述 补充 链表顺序
First-fit 从头开始扫描，选择大小足够的第一个可用块 最简单有效，低地址小分区多，高地址预留大分区，开销大 地址递增
Next-fit 从上一次放置的位置开始，选择下一个可用块 空闲分区均匀，减少开销，但是缺乏大的分区 地址递增循环
Best-fit 能满足要求、又是最小的分区 性能较差，小分区多，解决大进程分配 分区递增
Worst-fit 检查第一个是否可用 小空间少，大空间不会大，中小型作业 分区递减

比较

  • 最优适应分配算法的主存利用率最好 ，但某些情况下，最坏更好。
  • 如果空闲区按从小到大排列，则最先适应分配算法等于最优适应分配算法，反之等于最坏。
    • 这里是因为最先适应按照地址链接，最优按照分区大小，如果分区大小递增，相当于从低地址链接到高地址。
  • 兄弟：依据自己的block大小 232^323 ，然后从右往左第四位翻转即可

Pros&Cons

  • Simple，Efficient in relocation, easy in protection
  • Inefficient in the use of memory ，Contiguous memory placement, difficulty for sharing ，Linear address space—inconvenient in large-scaleprogramming and large project development 大规模变成的线性地址空间不好

Managing Insufficient Memory 内存不足

内存紧凑：真的内存不足吗?

  • 把分散于整个内存中的空闲块合并成为一个单一的大空闲块；
  • 数据先读出内存， 再写入，费时、开销大。 几乎没有现代 OS 实现了内存紧凑！

交换：新的进程真的需要那么多内存吗

资源浪费现象：阻塞进程占用大量内存，外存中的等待作业因无内存不能调入运行 。将内存中暂时不能运行的进程和程序数据调出外存，调入可运行的程序。

外存

  • 文件区：目标是提高文件存储空间利用率故非连续分配
  • 交换区：换出的进程，提高换入换出速度故连续分配

数据结构：交换区的空闲块使用空闲区表或空闲区链（交换区首址和大小）

分配：动态分配类似，首次，循环或者最佳适应

进程的换出

  • 进程创建子进程要空间
  • 阻塞且优先级低
  • 启动磁盘传送到磁盘swap区，未出错则回收内存，修改PCB

换入

  • 系统定时查看进程状态，找出就绪且换出时间最久的作为换入进程
  • 申请，不成功则再换出进程，申请换入

覆盖：内存中的进程都有用吗

  • 将进程所需的内存空间划分成一个固定区和多个覆盖区
    • 主程序放固定区
    • 依次 调用的子程序则放在 同一个覆盖区，即一次性
  • 操作系统提供 Overlay 的 System Call，供程序员在跳转到子程序之前调用

早期扩大内存容量的技术，在单道连续存储管理中使用，覆盖顺序结构由程序员规定。即选择一个更大的分支进行空间分配，并由程序员告知层次信息。

交换&覆盖

  • 程序员透明和不透明
  • 交换进程之间，而覆盖是一段进程
  • 现代OS交换技术仍有较强生命力

Virtual Memory Management 虚拟内存管理

背景：

  • 实内存不能运行比内存大的程序
  • 进程全部装入内存浪费空间。单控制流进程需要较少部分在内存，而多控制流则需要较多。

虚拟内存允许将大逻辑地址空间映射到小的物理内存

Non-continuous Memory Allocation andMemory Address Space

  • 虚存使进程部分装入内存，部分或全部装入外存，运行时访问外存则动态调入
  • 但是，内外存间的调动会十分频繁 。
  • 并不强调内存的连续性

分段：程序是一些段的结合，段是一个逻辑单位比如主函数，局部变量，符号表等。

  • 分段易于实现段的共享和安全保护，但是大小不一管理困难，后续引入段页式。

Paging 分页

分页也有实存模式，如果能够swap出内存，成为虚内存模式。同样的，虚拟内存将分页的优点发挥到最大。

晚于分段的出现，但普遍应用，因为简单且空块少。

概念 解释 补充
页面（page）/页（进程） 逻辑地址空间分成若干个大小相等的片 编号从 0 开始
页框（Frame)/物理块（内存） 把内存空间分成与页面相同大小的若干个存储块，称为或页框 进程ID+页面号=页框号
页内碎片 进程的最后一页经常装不满一块而形成了不可利用的碎片
页表 进程的每一页分散在内存，为进程建立页表 页号→页框号**PCB中存放页表地址PTBR **
页面大小 太小碎片减小但每个进程占用更多页面，页表过长，内存过大，页面换入换出频繁。过大碎片大，利用率低。 硬件决定（机器地址结构），2的幂次
页内偏移 页号和偏移量

逻辑和物理地址的转化

  • 逻辑地址和电脑位数有关，比如2322^{32}232 字节，包括逻辑页号和页内地址
  • 物理地址是页框号和页内地址
  • 页面编号从0开始，可以直接通过移位操作做除法，高位是页号，从0开始，低位是偏移量
  • 右侧的01标志是有效位，因为擦除并不是设置为NULL，可能只是覆盖。

分区过大->分页?

MMU：逻辑到物理的硬件加速

  1. 基本地址变换，只实现了「页号→页框号」

  2. 增加虚拟内存支持：「页号→页框号」有可能会失败，引入「缺页中断」

    • 缺页，MMU查询页表失败。
    • MMU产生中断信号，有中断向量，将OS唤醒，查找中断向量表
    • OS将block进程
    • OS进行io，找硬盘。IO操作在后台进行，CPU让位给下一个进程。
    • IO完成后硬件中断，唤醒OS，
    • OS修改页表，将进程放入ready，然后进行调度

    缺页中断和io中断导致内核态和用户态切换，由寄存器到内存到硬盘的速度降低

    本身我们从ptbr找页表就多做了一次查找

  3. 页表增加高速缓存 TLB：「读页表 + 读数据」涉及两次内存访问，为页表建立缓存，去除前一次内存访问；

    • 减少内存访问，将已经查过的表项先保存，先在寄存器组中查看，此时页表并不是一个连续的状态，利用寄存器本身的小处理器实现并行，同时查看（硬件处理），即TLB（快表或联想寄存器组）。贵且耗电（散热），故小规模，更新策略如果是先进先出，串行程序很好，但是如果是循环则over，另外策略是存一个符号表示最近访问

      期望百分之九十在这个cache，百分之九十九并不用查页表（页表在内存里，还要再访问内存数据，一共查两次）

页表表项

地址变换机构的实现

主要是页表的位置和从虚拟地址查找到内存地址的方案

❖ 缓存 （ Caching）：空间换时间
❖ 间接（ Indirection）访问：时间换空间，可以解决大部分的问题但是开销大啊

  • 页表都放在内存中，两次访问

  • 部分页表在高速缓存，快表TLB

    • TLB是快表，寄存器组，其实和上面那个重复了
    • 基于局部性原理保证TLB的命中率

  • 虚拟页式存储管理流程

    • 首先进行越界中断
    • 然后查找快表，把自己写进去
    • 缺页中断，保存现场尤其是PC和状态，去外存找缺页
    • 找内存里有无空余地区，无则将受害页踢出去，选择策略就是FIFO等
    • 删除对象如果删除位M=1，则将修改内容读入外存后删除，然后插入

多级页表

多级页表类似多维数组，元素单位是表项，一级页表是连续存储所有表项。也就是说将表项分成页面

  • 二级页表首先根据 “伪表项”找到存放表项的页面，然后根据页内偏移找到真实的表项，然后根据表项找到物理内存地址

  • 如下图的例子，我们首先计算虚拟地址允许的最多页面数是2202^{20}220，对应需要页表表项内存是220∗22=2222^{20}*2^2=2^{22}220∗22=222。也就是说，我们的页表空间最少是2222^{22}222

  • 将页表分页，得到第一级页表，需要2102^{10}210个页面，将这些页面的地址记录在二级页表中

  • 计算问题：第二级是页面大小除以entry即可得到一个页里面能够对应多少下级页。按照上面的计算得到的是逻辑地址是否完全用上

    使用数组的理解方式，210×210=2202^{10}\times2^{10}=2^{20}210×210=220个页面。

换个例子：64位机 4byte entry 4M大小页面，4M-> 22 偏移，直接得到 20（2） 20 20 22的地址标识

硬要分析，偏移22，一共有64-22=42 即2^42 个页面（1,2,3级之和）， 42+2-22=22 个表项页面（2,3级之和），22+2-22=2（第3级页面）

倒排页表

  • 传统页表面向进程空间，即逻辑地址空间，但是一个很空的表
  • 倒排页表面向内存空间(物理地址空间) ，每个内存页框一个表项，大小固定， 整个系统共用一个页表
    • 查找不方便。把倒排页表放入快表中进行并行查找，但是物理内存不能大

使用哈希函数的倒排页表？？？

  • 本质上是哈希页表，表长是页框数
  • 哈希冲突：“本地拉链法”，本地一定有空余位置
  • 表只需要n个进程的大小，可以把表放入内存，因为整齐可查，如果表放不下，则分成两块，类似多级页表，但使用的哈希后的数字作为前面的字符串

分页的共享与保护

  • 存储保护
    • 越界保护：设置页表的长度寄存器防止查表越界
    • 操作访问保护：页表项设置存储保护域，说明访问权限
  • 分页共享：分页未考虑逻辑意义，硬性共享可能导致逻辑的破坏

Segmentation 分段

概述：

  • 段是逻辑意义上的模块，比如主函数，过程，函数，对象，堆栈，数组等，各自占用连续分区，从零开始
  • 方便共享，保护
  • 碎片多，不好管理，数据也比较混乱（动态分配，堆栈对空间的利用程度取决于动态分配 ）

目的

  • 编程方便：程序按逻辑分段后可以直接根据段名加偏移访问逻辑地址
  • 信息共享与保护：逻辑意义上的模块
  • 简化对不断增长的数据结构处理
  • 动态增长：对于数据段等使用过程中增长的情况
  • 动态链接：可以将段作为管理单位

段表

  • 每个分段分配连续分区，进程的各个端离散装入不同分区，段表用来映射内存基址

  • 由于段长不同，需要另外存储，为实现简单，常用段号代替段名

  • 分段在编译时由编译程序完成。

分段下的信息分享和保护

  • 可重入代码 (Reentrant Code) 又称为 “纯代码”(Pure Code) 是一种允许多个进程同时访问的代码。可重入代码是一种不允许任何进程对它进行修改的代码
    • 程序在执行时可能改变的部分，拷贝到该局部数据区。
    • 程序执行时，只对该数据区 (属于该进程私有) 中的内容进行修改，而不 去改变共享的代码。

保护

  • 地址越界保护（上、下界寄存器或基址 + 限长寄存器）
  • 存储保护键法（每个存储块分配一个单独的保护键，每个进入系统的作业也赋予一保护键，相当于钥匙。）
  • 存取控制

段页式

  • 先将用户程序分成若干段，再把每段分成若干页。
  • 逻辑地址由3部分组成：段号、段内页号、页内地址

Operating System Algorithms for Virtual Memory 应用程序生命周期与虚拟内存管理

概览

  • 启动
    • Load Control 装入控制( 控制内存中的进程数目/并发度) ，复杂
      • 驻留进程过少则进程可能阻塞大反之频繁缺页中断
    • Fetch Policy ：
      • Prepaging：预调页策略 ，以预测为基础，在连续分配时，一次调入若干相邻的页，难以实现
      • Demand paging：请求调页策略 ，当进程在运行中需要访问某部分程序和数据时，若发现其所在的页面不在内存，便提出请求，由OS将其所需页面调入内存
    • Placement Policy： 不重要，取来放哪
  • 驻留在内存：resident set 驻留集管理
  • swap出去
    • Resident Set Management 驻留集管理
      • 每个进程的页面分配多少页框
      • 页框不够，新的页面又一定要调入内存时，决定在内存的哪一个范围内选取被置换出去的页面
    • Replacement Policy ：置换策略
      • 当页框不够，新的页面又一定要调入内存时，决定被置换出去的页面
  • over
    • cleanning policy，也分为demand和precleaning，不过不重要

置换策略

页面置换策略也可用于段置换、 TLB置换，内存中希望有很多进程以减小上下文切换的开销但是缺页中断会更频繁

  • 最优置换策略（ OPT）：无法实现的最优策略

    • 选择未来最长时间内不再使用的页面来置换
    • 理论上缺页率最低

  • 先进先出策略（ FIFO）： 最直观也是效果最差的策略

  • 访问并不算在里面，即使你天天用，但是来得早也得滚蛋

  • 最近最旧未使用策略（ LRU）:可行的最优策略，但硬件代价高，频率一致就比时间

  • 需要对每一页的使用情况跟踪记录，系统开销较大

  • 可以借助“栈”来完成：栈顶始终是最新的，栈底是最近最久未使用的页面号

  • 时钟策略（ CLOCK）:LRU的简化版本

  • 只需为每页设置一位访问位，再将内存中的所有页面都通过链接指针链接成一个循环队列

  • 当某页被访问时，其访问位被置1

驻留集

固定：平均或者按照进程比例或者优先级等算法

可变：多了可能效果很好但是浪费了就减少

窗口大小决定工作集大小，2-5计算下来都是2-3

抖动是换出去换回来，是一个无用的IO开销过程

可变，频率，求余？只要有需求就去做

预调用策略在初始化的时候常用，即相比之前的全部空白而言，随便装，其他几乎不用

我们几乎忽略了访问快表的时间，因为非常小

belady 增加叶框反而增加了缺页概率，非考点

clock：没有考虑修改，故需要另外存储，我们仅考虑一个访问位，则只需要设置一个修改位即可，倾向于没修改的

低效率置换，最差：既访问了也修改了

局部性导致的故障值很大
