Xv6操作系统接口

内核

xv6 提供了传统的内核概念

内核kernel是os的基本部分， 为众多应用程序提供对计算机硬件的安全访问的一部分软件，这种访问是有限的，并且内核决定一个程序在什么时候对某部分硬件操作多长时间。

  • 进程通过系统调用使用内核服务。系统调用会进入内核，让内核执行服务然后返回。所以进程总是在用户空间和内核空间之间交替运行。

  • 系统调用 system call ：进程（程序）进行系统调用，系统调用进入内核执行命令

进程

结构体

xv6无TCB结构，即无进程抽象，使用proc的结构表示线程，最重要的是：页表（内存管理），内核栈(kstack)，当前运行状态（仅内核可见）。

这里不用上下文切换，而是运行线程（线程）来执行进程的指令，可以被挂起或运行，来表示进程的切换。

线程的大多数状态（局部变量和函数调用的返回地址）都保存在线程的栈上

进程的分时特性：进程在可用 CPU 之间不断切换，当一个进程不在执行时，xv6 保存它的 CPU 寄存器，当他们再次被执行时恢复这些寄存器的值。内核将每个进程和一个 pid 关联。

用户栈和内核栈（p->kstack）

  • 进程运行用户指令时，只有其用户栈被使用，其内核栈则是空的

  • 当进程（通过系统调用或中断）进入内核时，内核代码就在进程的内核栈中执行，用户栈仍然保存着数据，只是暂时处于不活跃状态

    处理器转入内核栈中，提升硬件的特权级，然后运行系统调用对应的内核代码。当系统调用完成时，又从内核空间回到用户空间：降低硬件特权级，转入用户栈，恢复执行系统调用指令后面的那条用户指令。线程可以在内核中“阻塞”，等待 I/O, 在 I/O 结束后再恢复运行。

  • 进程的线程交替地使用着用户栈和内核栈。内核栈是用户代码无法使用的，这样即使一个进程破坏了自己的用户栈，内核也能保持运行。

状态

p->state 指示了进程的状态：新建、准备运行、运行、等待 I/O 或退出状态中。

页表

xv6使用页表（由硬件实现）来为每个进程提供其独有的地址空间。

为每个进程维护了不同的页表，合理地定义进程的地址空间了

p->pgdir 以 x86 硬件要求的格式保存了进程的页表。xv6 让分页硬件在进程运行时使用 p->pgdir。进程的页表还记录了保存进程内存的物理页的地址。

  • 一个xv6进程由两部分组成，一部分是用户内存空间（指令、数据、栈），另一部分是仅对内核可见的进程状态。（内核的数据映射）？？

一片地址空间包含了从虚拟地址0开始的用户内存。它的地址最低处放置进程的指令，接下来则是全局变量，栈区，以及一个用户可按需拓展的**“堆”**区（malloc 用）

将虚拟地址翻译为物理地址

  • 内核的指令和数据也会被进程映射到每个进程的地址空间中，进程使用系统调用时，系统调用实际上会在进程地址空间中的内核区域执行，可以让内核的系统调用代码可以直接指向用户内存
  • xv6 将内核映射到了地址空间的高地址处，即从 0x80100000 开始（给用户留内存）

文件描述符

文件描述符是一个强大的抽象，将连接的细节隐藏：一个进程向描述符1写出，它有可能是写到一份文件，一个设备（如控制台），或一个管道。

打开现存文件或新建文件时，内核会返回一个文件描述符。读写文件也需要使用文件描述符来指定待读写的文件

  • 0: 标准输入、1: 标准输出、2: 标准错误

常称为“文件”，一个整数，（看上去像是字节流？），它代表了一个进程可以读写的被内核管理的对象，进程可以通过多种方法获得一个文件描述符，如打开文件、目录、设备或者创建一个管道、或者复制已经存在的文件描述符。

  • 每个进程都有一个从0开始的文件描述符空间 ：xv6将文件描述符作为进程表的索引

  • 系统调用

    • read和write从文件描述符所指的文件中读或者写n个字节。read从fd中读最多n个字节，将它们拷贝到buffer中，然后返回读出的字节数。每一个指向文件的文件描述符都和一个偏移量有关。

    • linux中一切皆文件，分普通和目录文件等，目录文件包含了哥哥文件和执行文件的指针

    • 任何硬件设备都必须跟根目录下某一目录执行挂载让文件系统在os中可用，mount命令操作，否则无法使用

    • cat的实现：将数据从标准输入复制到标准输出，自己并不知道两个具体对象

    • close 会释放一个文件描述符，使得它未来可以被 open, pipe, dup 等调用重用

    • 一个新分配的文件描述符永远都是当前进程的最小的未被使用的文件描述符。

  • I/O重定向实现：将一个文件描述符标识的文件替换为另一个文件

    • fork 会复制父进程的文件描述符和内存
    • exec 会替换调用它的进程的内存但是会保留它的文件描述符表。
    • 单独调用的原因： shell 可以在子进程执行指定程序之前对子进程进行修改。

    举例： 在shell中输入指令cat<input.txt，shell会执行代码

    • open 会使用0作为新打开的文件 input.txt的文件描述符（因为0是 open 执行时的最小可用文件描述符）
    • cat 就会在标准输入指向 input.txt 的情况下运行，实现新建文件的功能

  • 父子进程与文件描述符

    • fork 复制文件描述符，但是每个文件的偏移共享

    • dup 复制一个已有的文件描述符，返回一个指向同一个输入/输出对象的新描述符，同样共享文件进度

文件系统

xv6 关于文件系统的操作都被实现为用户程序，诸如 mkdir，ln，rm 等等。这种设计允许任何人都可以通过用户命令拓展 shell

例外：cd ，不能作为普通命令执行

Unix 时代的其他系统的设计都将这样的命令内置在了 shell 中，而 shell 又是内置在内核中的。

如果 cd 作为一个普通命令执行，那么 shell 就会 fork 一个子进程，而子进程会运行 cd，cd 只会改变子进程的当前工作目录。父进程的工作目录保持原样。

文件与目录

  • 文件就是一个简单的字节数组

  • 不从 / 开始的目录表示的是相对调用进程当前目录的目录

  • chdir调用进程的当前目录

实验准备

系统调用的过程

  1. 用户态运行程序，涉及在user.h声明过的system call ，转到usys.S文件，通过ecall进入中断处理函数

    所有user.h 函数的声明都是在usys.S文件中使用risc-V汇编实现，汇编会调用syscall.h中的编号

    ecall跳转到内核栈，并且保存现场（栈指针sp，程序计数器pc等），结束系统调用的时候会回到当前状态

  2. kernel 内核态进入文件trampoline.S ，uservec跳转到内核指定的中断判断函数

  3. trap.c文件中，函数usertrap 判断如果是system call执行响应函数syscall()

  4. syscall.c文件中 syscall（）函数进入相应处理函数

    syscall.c文件中会有 外部函数服务的声明以及指向该函数的指针定义，syscall（）根据a7获得函数指针，调用函数。

  5. sysproc.c中实现sys_xxx的具体函数，如果有参数，需从中断帧保存寄存器中能够取得

    上一步调用的函数都在sysproc.c中实现。

    调用完成之后，系统调用的返回值会在返回用户态时，被赋到 a0 寄存器上

  6. proc.c 真正进行创建进程

结果分析

  1. 在命令行输入trace 32 grep hello README指令之后，根据trace.c，
    • 会先调用trace()，然后再执行grep命令。
      • trace()，在usys.S中转入内核态后，将sys_trace（void）的编号保存在a7中，在syscall()中利用a7运行sys_trace，这个函数中我们使用argint()获得第一个参数32，并且写入myproc()->mask，此时num的值我们可以看到是调用sys_trace的调用号22，1<<num&32=0即不打印
    • 在执行grep命令，即此时的a7是sys_read，后面num的是调用sys_trace的调用号5，故打印
    • 由于25=322^5=3225=32，故只有num<=5的系统调用会被追踪，具体到grep中只有read会被打印
  2. 在命令行输入trace 2147483647 grep hello README
    • 与上面类似，trace()实现myproc()->mask的值为2147483647
    • trace调用号为22=num，故1<<num &2147483647 不等于零，打印
    • exec调用号是 2=num ，1<<num &2147483647 不等于零，打印
    • 由于231=21474836482^{31}=2147483648231=2147483648，故num<=30的均可以调用，也就是目前所有的系统调用都可以被打印
  3. 在命令行输入 grep hello README
    • 不执行调用程序，故无输出
  4. 在命令行输入trace 2 usertests forkforkfork
    • 与上面类似，trace()实现myproc()->mask的值为2
    • 在所有的系统调用中，只有fork会被打印

实验题目：Traps

实验目的：

实验准备

函数调用

  • 由于函数调用和栈的相关性，可以将每次函数调用抽象为栈中的元素，即栈帧（stake frame），每次都执行栈顶元素。
  • 编译器在每一个栈帧中放入一个调用者地址的 frame pointer。通过这些指针来遍历栈，从而打印每个栈帧的返回地址
  • gcc编译器将目前的栈帧存放在s0，需要加入函数获取
  • 返回地址储存在距离帧栈顶部（-8）偏移处， 指向上一个帧栈的指针储存在帧栈顶部（-16）偏移处
  • xv6 为每个栈分配一页的大小，并且栈底地址是 PGSIZE对齐的，可以使用 PGROUNDDOWN(fp)和 PGROUNDUP(fp) 来计算栈顶和栈底的地址。

trap

  • 用户态切换到内核态的有两种情况：一种是系统调用（syscall），另一种是异常（trap）或中断（interruption）
  • trap是传统的Unix术语，由在CPU上运行的当前进程导致，和中断不同，但是都依赖于相同的硬件机制
  • 从用户态转到内核态的时候，都相当于执行了一个中断响应的过程
    • 保存32个用户寄存器和pc
    • 将CPU的字段改为内核态（硬件标志位）
    • SATP指向kernel page table
    • 堆栈寄存器指向位于内核的一个地址

xv6中的trap

  • 用户态在进入内核态的时候调用了一个entry
    • ecall将CPU中的mode标志位设置为supervisor，并且设置pc的值为STVEC寄存器的值
    • 内核会将trampoline page的地址存在STVEC寄存器中
  • kernel/trampoline.S从用户空间到内核空间并返回的程序集，其中struct trapframe *trapframe; // data page for trampoline.S，即寄存器保存数据的位置
  • kernel/trap.c: 处理所有中断的代码

      all_thread数组是未初始化的全局变量，初始值默认为0

    • 修改两个寄存器，另外切换的时候这俩也是核心

      • ra寄存器：return address ，设置为func的地址，线程执行函数func

      • sp寄存器：初始化栈指针，这是因为线程的栈结构独立，运行函数要在线程自己的栈上，设置为stack数组的尾部，（栈由高到低增长）。

  4. thread_schedule()的更改

    • 首先我们找到kernel/swtch.S/swtch()函数，因为我们自己设置的th_context是直接赋值context的内容的，寄存器一致，故直接复用

    • 找到shred()的调用语句

(三) Barrier

实验要求

实现满足线程同步的 barrier

实验准备

  • 包括之前的信号量的原语，我们增加新的原语

实验内容

前置

  • 运行barrier程序，要求多线程执行到同一位置之后继续运行

  • 即不满足该性质

思路

  • 观察实验文件notxv6/barrier.c可以知道，函数依据我们传入的参数数目创建线程，然后每个线程会执行20000barrier，要求执行barrier()的时候有阻塞，所有线程都执行到barrier才能解除阻塞。

  • 根据提示，我们使用条件变量pthread_cond_t实现，观察的结构体，包含互斥锁和同步锁，nthread用于记录已经到达的线程数量设置为传入参数，round记录barrier的轮数，初始化为0，锁也在调用 barrier_init()完成了初始化

步骤

  1. 按照之前的分析直接更改barrier()

    • 对于条件变量的操作，需要保证互斥，即直接加锁
    • 然后进行判断是否所有的进程都进入函数，如果没有，则加锁，反之释放，同时将到达线程数目nthread重新设置为零，round更新加一表示以及完成了一轮

思考题

  1. Uthread: switching between threads:thread_switch needs to save/restore only the callee-save registers. Why?

    我们是模仿内核进程切换的函数，在内核进程切换时，xv6的调用规则是将非calllee-save的寄存器内容保存在线程的堆栈中，在恢复的时候直接恢复对应寄存器的值，而剩余的寄存器也就是calllee-save寄存器是由被调用函数即thread_switch()保存，返回的时候已经丢失，需要额外保存这些寄存器的内容

  2. Using threads:Why are there missing keys with 2 threads, but not with 1 thread? Identify a sequence of events with 2 threads that can lead to a key being missing.

    ph.c实现了一个哈希表，如果多个key映射到同一个bucket，哈希表的做法是在bucket后面挂一串链表，每个value都占一个链表节点，对应一个key，此时在insert()函数会出现race condition。（insert参数传递见补充的注释）

    而在哈希表的put函数和get函数中，都没有设置任何线程安全机制，故多线程插入的时候可能会覆盖数据从而导致数据丢失。

    具体过程：

    • 设置哈希表桶值为NBUCKET，哈希函数为i=key%NBUCKET，数组bucket是一个指针数组，存储链表头指针。

    • 线程A和B同时put，如果两个哈希值相同，则会插入到一个桶中，并且下面插入函数的后两个参数相同，假设均运行到insert处:

    • 设A先插入完成，此时数组中对应头指针是新的节点。而B插入的参数并没有改变，也就是说，第三个参数对应数组元素的地址没有问题，但是第四个参数却是旧的头指针。

    • 运行函数中，B的节点会将next=旧链表头，然后更新数组中对应链表头指针为自己，直接覆盖了A的操作。

实验感想

  • 在第一个实验中，一开始传参的时候并没有添加强制类型转化，则会出现传参的报错，需要加上（unit64）的转化
  • 实验难度适中，能够很好的帮我们理解有关进程调用的原理
